# AXIOMA CONTEXT

## PRODUCT DEFINITION
Axioma is a reasoning diagnostic platform.

It does NOT evaluate answers.
It identifies where reasoning breaks and fixes it.

Focus: mathematical thinking, not memorization.

---

## CORE PRINCIPLE

Gaps = conceptual misunderstandings  
Signals = answer patterns indicating gaps  

Rule:
If fixing requires new theory → GAP  
If not → SIGNAL  

Gaps must remain small and stable.
Do NOT create micro-gaps.

---

## CORE LOOP

Diagnostic → Theory → Training → Diagnostic (reassessment)

This loop is the product.
Do not break or bypass it.

---

## MODULES

### Diagnostic
- Detects reasoning gaps via patterns (not single mistakes)
- Uses multiple signals per gap
- Outputs gap profile (not score)

---

### Theory
- Explains ONLY the section where the gap exists
- Not full topic explanations
- Includes mini-checks and notes

---

### Training (single page, 2 modes)

Targeted mode:
- Triggered from a detected gap
- 8–10 focused tasks
- Purpose: repair specific reasoning gap

Free mode:
- No gap filter
- General practice

---

### Mastery (optional)
- Validates understanding across gaps
- Not absolute proof, only strong signal

---

## WHAT DOES NOT EXIST

- No Homework system
- No redundant pages
- No exam simulation

---

## CURRENT GAP MODEL (Quadratics)

Core gaps include:
- Conceptual understanding (e.g. discriminant)
- Transformations
- Logic errors
- Missing cases
- Pattern misuse

Additional topic gaps:

q-adaptive:
- Failure to recognize same structure in different forms

q-relational:
- Failure to translate between representations (graph, algebra, verbal)

These are conceptual, not micro-errors.

---

## DETECTION MODEL

- Based on answer patterns (signals)
- NOT static answer IDs
- Compare given answer vs correct answer
- Accumulates evidence across questions

---

## PRODUCT GOAL

Improve reasoning, not performance.

Success = user:
- understands concepts
- transfers knowledge to new forms
- makes fewer conceptual errors over time

---

## EXPERIMENT CONTEXT

Small pilot (5–8 users per group):
- Axioma group vs control group
- Pre-test → Post-test → Delayed test

Measure:
- score
- error types
- gap reduction

---

## DEVELOPMENT RULES

1. Do not overcomplicate gaps
2. Do not add features before validating core loop
3. Every feature must support the loop
4. Keep UI minimal and focused
5. Prioritize clarity over completeness

---

## DO NOT

- Suggest rewriting working systems
- Add unnecessary abstraction
- Split gaps into many small categories
- Break the core loop
- Mix philosophy discussion with implementation

---

## WHEN UNSURE

Default to:
- simpler system
- fewer gaps
- clearer flow
